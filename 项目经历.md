# 2.在这个项目中的遇到的困难是什么，怎么解决的

1.**我遇到的一个问题是，做高并发查询的时候会导致数据库压力过大。**

最开始我用的是MySQL主从架构做读写分离，

我设计完这种架构之后，我查阅了资料发现模糊查询可能会产生性能问题，

我这个评价系统支持用户按关键词搜索（比如"质量"这样的关键字），

这种需求在SQL里需要用 `LIKE '%关键词%'` 实现的，

**然后，我设计了一个实验想要验证这个猜想** ：

使用Go脚本批量插入了10万条评价数据，

每条评价包含：用户ID、商家ID、评价内容（100-200字）、评分等，

模拟了3个商家，每个商家3万+评价，

插入了2万条回复数据和5000条申诉数据，

数据规模虽然不如生产环境，但我认为能够暴露一些问题。

然后我开启了MySQL的慢查询日志，long_query_time =0.5 超过500ms就记录，

我分别试了三种查询方式：

1. 精准查询（只用索引字段）：
   SELECT * FROM review_info WHERE store_id = 10001 LIMIT 10;
2. 前缀模糊查询：
   SELECT * FROM review_info WHERE content LIKE '质量%';
3. 全模糊查询：
   SELECT * FROM review_info WHERE content LIKE '%质量%';
   → 响应时间：600-1200ms（全表扫描）

对于全模糊查询然后我通过EXPLAIN分析执行计划，

-type: ALL（全表扫描）

-rows: 100000（扫描10万行）

发现果然是LIKE '%关键词%'这种模糊查询导致索引失效，变成了全表扫描。

**而且另一个问题是商家查看评价时需要关联三张表**（评价表+回复表+申诉表），

这样也会给从库增加负担。

（压测）

*我用JMeter做了简单的压测：*

- *并发用户：100（模拟100个商家同时查询）*
- *每个用户执行10次带模糊搜索的查询*

*结果：*

- *平均响应时间：1.2秒，和我之前explain发现的响应时间差不多，*
- *我的电脑CPU使用率飙到60%（只是i5处理器），*
- *MySQL慢查询日志里记录了大量超过500ms的查询*

*虽然这不是生产级别的压测，但已经能明显看出性能问题。*

**解决方案！！！！！！！！！**

所以我参考了一些开源社区项目的架构方案，采用进阶版的CQRS读写分离架构：
写操作：继续用MySQL保证事务一致性
 读操作：

- 热数据（高频查询）：走Redis缓存，响应时间降到毫秒级
- 冷数据（复杂查询）：走Elasticsearch，利用倒排索引做全文检索
  数据同步是通过Canal监听MySQL binlog → Kafka → review-job消费写入ES

（压测）

**然后优化后我再次用JMeter进行相同测试，我记得当时ES响应时间100ms左右，**

CPU使用率也降了很多，百分之多少我忘记了。这就说明这个之前的方案在我这个项目场景下，

尤其是高并发下确实存在问题。

**2.还有一个问题是考虑到缓存击穿的问题。**

假设有1万个并发请求查同一个商家的评价，

Redis中恰好没有这个key（缓存过期或冷启动），

1万个请求同时打到ES，导致ES压力骤增甚至崩溃。

所以我采用的解决方案就是使用Singleflight合并相同请求：

假设1万个相同key的并发请求，那么只有1个请求真正查ES，

其他9999个请求等待并复用第一个请求的结果。ES压力就会降低。

**3.还有就是Mysql和ES的数据同步延迟的问题。**

用户在review-service创建了一条评价（写入MySQL），
Canal监听binlog → Kafka → review-job消费 → 写入ES，
这个链路可能需要几百毫秒甚至几秒，
如果用户立刻去查询，ES中还没有数据，产生"数据丢失"的错觉。

所以我当时采用的方案是，
 创建评价后，返回reviewID给用户
 用户如果刚一评价完就立刻查询详情时，

优先查MySQL（用reviewID精确查询，有索引很快）
 列表查询才走ES（用户不会立刻去刷新列表，有时间让数据同步）

如果用户过一段时间再次查看评价的话，这时候ES也有数据了，

而且别人可能期间也查看过这个用户评价的内容，那么redis可能也有热点数据了。

我后续的话打算做一个长期优化策略：
监控Kafka消费延迟，设置告警信息lag，

如果 lag 一直在增长，说明有积压，
增加review-job的消费者数量，可以让延迟时间更短

**4.还有就是分布式ID的生成问题**

这几个微服务都需要生成全局唯一的ID（reviewID），

如果用MySQL自增ID，分库分表后会重复，

如果用UUID，36字节太长，存储浪费，

而且无序不利于索引，不适合做主键索引，

"UUID完全无序，这就会导致MySQL B+树索引频繁页分裂，
    而雪花ID是递增的，新数据总是追加到索引末尾，
    避免了页分裂问题"

后来我采用了雪花算法生成ID的方法解决了。

**5.还有一些比较小的问题**

第一次部署Canal时遇到的坑：
Canal连不上MySQL：权限问题，
 Canal监听不到binlog：MySQL的binlog没开启，
 Kafka消息格式解析错误：不熟悉Canal的JSON格式，

# 3.什么是CQRS架构？

cqrs一般叫：命令查询职责分离。
它的核心是把‘写’和‘读’拆成两套职责：
写侧只负责业务规则和一致性，读侧只负责查询效率和展示效果。
所以可以理解成，我把写和查拆开做了。
写这边就老老实实保证事务，查这边就专门做性能。
两边数据不是硬绑在一起的，写完之后会有个同步过程，所以偶尔会有一点点延迟。
但换来的好处是，读请求再大也不容易把写链路拖垮。

结合我这个评价系统来说：
写操作，比如用户发评价、商家回复，我走写侧服务，先落 MySQL 主库，保证事务正确；
读操作，比如评价列表、按条件筛选、统计聚合，我走读侧，主要查 Redis/ES，减少主库压力。
然后我用异步链路把写库数据同步到读模型，这样能支撑读多写少场景下的高并发查询。

# 4.为什么要设计redis+singleflight这种模式

`singleflight` 是 Go 里的并发请求合并组件。
它主要解决的是**短时间内同一个 key 的重复回源**问题，常用于缓存击穿场景。

如果只有 Redis + ES，在高并发下某个热点 key 过期时，

可能会有大量请求同时 miss，然后一起打到 ES，导致 ES 压力暴涨甚至雪崩。

加上 `singleflight` 后，同一个 key 在同一时刻只会放行一个请求去回源（查 ES）；

其余请求会等待并复用这个请求的结果。
回源成功后再写回 Redis，后续请求直接走缓存。

# 5.这个项目的业务流程是什么

**我的评价系统涉及三端（C端用户、B端商家、O端运营），**

用户下单购物
→ 订单完成后可以评价
→ 填写评分（1-5星）、评价内容、上传图片/视频
→ 提交评价
→ 评价进入待审核状态（status=10）（可以被查到）

运营人员登录后台
→ 查看待审核评价列表
→ 审核内容是否违规（广告、辱骂等）
→ 审核通过（status=20）或不通过隐藏（status=40），

并且标记这个评论为已审核
→ 审核通过的评价继续展示在商品详情页

商家登录后台
→ 查看自己商品的评价列表
→ 对评价进行回复（比如感谢好评、解释问题）
→ 如果认为评价不实，可以发起申诉

运营审核商家的申诉
→ 申诉通过：隐藏该评价（status=40）
→ 申诉拒绝：评价继续展示，并且将这个回复标记已审核

# 6.用户创建评价的详细流程是什么，又哪些校验

用户创建评价的流程有参数校验、业务校验和数据持久化的过程

**1.参数校验（service层），**使用了Proto的validate自动校验，比如userID必须大于0，评分要在1-5之间，

校验失败直接返回400错误。

2.**业务校验（Biz层）**，例如检查订单是否已评价（防止重复评价），

3.**持久化（Data层），**写入MySQL，成功后自动产生Binlog，Canal监听到变更，发送到Kafka，

review-job消费并同步到ES。.

# **7.如何保证数据一致性？**

**业务上：**

比如商家回复评价涉及两张表的更新，我用数据库事务保证一致性。

因为我这个项目针对这个问题做了两个事情

1.商家回复需要往回复表中插入一条回复记录

2.用户评价表的回复状态字段更新为1

如果第一步成功第二步失败了的话，就会出现：回复表有数据，

但评价表回复字段还是0，显示未回复，这样的话用户就看不到商家的回复了。

所以我就用了数据库的事务来保证这个事情，

首先要进行简单的业务校验，比如不能回复已经回复过的评价，

不能水平越权回复别人评价。

然后进行事务操作，先将回复记录插入回复表，失败自动回滚，

然后将回复的状态值插入评价表，失败也自动回滚。

**读链路上:**

我的项目场景：**用户创建评价 → 写入 MySQL → 同步到 ES，**

我用的是 **Canal + Kafka 方案，**

1. **首先是通过MySQL 的 binlog 来保证数据一致性**
   比如用户创建评价，写入 MySQL

   MySQL 会自动产生 binlog（数据变更日志）

   **只要 MySQL 事务提交成功，binlog 就一定会产生**
2. **Canal 监听 binlog 并发送到 Kafka**

   * Canal 伪装成 MySQL 的从库，实时读取 binlog
   * 把 binlog 变更发送到 Kafka
   * **Kafka 的持久化保证消息不丢失**
3. **消费者模块消费 Kafka 并写入 ES**

   * 消费者会 从 Kafka 拉取消息
   * 使用评价 ID 作为 ES 文档 ID 写入（保证幂等）
   * **即使重复消费，ES 也只有一份数据**

# 8.如何保证读写数据的一致性？

我项目里做法是分场景保证一致性: 用户看自己的评价走 MySQL 直查，（用userID的列表查询）

所以写完基本立刻可见；商家看列表走 Redis+ES，靠 Kafka 异步同步，

允许短暂延迟。也就是说，我把强一致留给关键路径，把高性能留给高并发查询路径。

这个思路和支付场景很像，用户完成支付等关键操作后，

必须先保证结果可见，然后再去做异步链路的数据扩散和查询优化。

# 8.ES的作用是什么？

在我的项目里，ES是评价搜索的核心引擎。

它负责复杂查询：按商家、关键词、评分区间、时间筛选、排序、分页、全文检索。

在评价系统里，用户常做的查询是：

* 按商家 ID 查评价
* 按关键词（如“服务好”“送货快”）做模糊搜索
* 按评分区间（4星以上）筛选
* 按时间排序、分页
* 按是否已回复、是否已审核等多条件过滤

MySQL 这类查询往往会走全表扫描，响应慢；

ES 用倒排索引+布尔查询可以在千万级数据上毫秒级返回。

因为mysql的innodb引擎处理索引的时候使用的b树，

他处理“like %关键词”这样的查询的时候，会导致索引失效，

要把每一行都读出来做匹配。而ES的**文档存储+**倒排索引：按“词”反向索引文档，

再结合 `score >= 4`等过滤。先找词，再确定文档” **，所以索引能够命中。**

# 9.ES为什么不能代替MySQL呢？

ES是搜索引擎，用于复杂检索；MySQL是主存数据库，

用于事务和一致性。ES不能代替MySQL，

因为它不支持完整的事务和强一致。所以一般推荐的架构是‘写MySQL，读ES’。

# 10.ES为什么不能用Redis替代呢（两者关系）？

**Redis = 内存KV缓存** ，只适合“按key拿值”，

不支持“按词模糊搜索+多条件过滤+排序”；

**ES = 搜索引擎** ，支持“倒排索引+布尔多条件+分页排序”，适合评价搜索；

所以：Redis可以 缓存热点结果（加速），但不能替代ES做复杂检索。

# 11.ES的查询结果是如何放在Redis的KV中的？

我的做法是把一次复杂查询的结果缓存成一个 KV。
key 用查询条件拼出来，value 存的是 ES 返回结果的 JSON。

具体流程是：先根据查询参数生成唯一 key，先查 Redis，命中就直接返回。
如果没命中，再去查 ES；拿到结果后，把整个 hits      序列化成 JSON 回填到 Redis。
我这边缓存 TTL 设的是 10 秒。
另外，为了防止并发场景下同一个 key 同时失效，

我用了 singleflight 合并请求，避免瞬时把 ES 打爆。

（`hits` 就是 ES 查出来的数据包，`total` 是总条数，`hits` 数组是当前页的具体记录。

`hits` 就是 ES 查询命中的结果集合。）

# 11.用户进行复杂查询走ES还是Redis？

1. 用户发起复杂查询（筛选+排序+分页）
2. 我先把查询参数规范化，拼成一个缓存 key，去查 Redis
3. 命中就直接返回；没命中就去 ES 执行真正的复杂检索，再把结果回填 Redis（短 TTL）
4. 所以 Redis 是“加速层”，ES 才是“检索层”

# 11.ES挂了怎么办？如何防止？

**产生的问题：**

ES是查询层，挂了后新写评价仍然写MySQL正常（写路径不依赖ES）。

查询层受影响：如果Redis命中还行，Redis Miss后ES不可用，必须回源MySQL查询。

挂了的结果就是数据不丢，只是查询会慢、复杂检索不可用。

容灾手段：

1. **降级策略**
   先查Redis缓存（热数据）

   Redis Miss再查ES（正常快速）

   ES不可用时降级到MySQL（兜底）
   这就是“Redis→ES→MySQL三层降级”，保证可用性，不会导致整个业务崩。
2. **集群副本机制**

   线上肯定不是单机 ES，都是集群加副本的。
   所以单个节点挂了，一般不会整个服务都挂，

   副本分片还能顶上去，查询还能继续。

   平时我们会盯监控，比如集群健康状态、节点是否掉线、

   有没有未分配分片、磁盘是不是快满了。
   一旦出问题，先做止血：重启故障节点，或者先扩容，把集群状态拉回来。
   然后再处理分片恢复和重平衡，保证集群稳定。

   如果这次故障导致数据有延迟或者不一致，我们会做补索引。
   增量不够就跑全量，从 MySQL 把评价数据重新同步到 ES。
   最后会跑一次 MySQL 和 ES 的对账脚本，确认数量和关键字段一致。

# 12.redis作用是什么？

Redis 在我项目里主要是做加速和保护后端。

redis我这里做了热点缓存。像高频的评价列表查询，我会先查 Redis，命中就直接返回。
redis还减轻后端压力，否则每个请求都打 ES 或 MySQL，

在高并发下很容易把存储层压满。Redis 走内存，响应会快很多。
他还做了抗并发的冲击，比如针对同一个 key 的并发请求，

我用 singleflight 合并，避免缓存失效瞬间大量请求同时穿透到 ES；同时也会缓存空结果，降低穿透风险。
然后就是做了短 TTL 策略。我这里缓存是短期热数据，过期快，

这样可以保证性能的同时尽量控制数据陈旧问题。

# 13.redis挂了怎么办？如何防治？

**产生问题：**

Redis挂了不会丢业务数据，也不会影响写入；只是查询回到ES读。

可能会对ES造成一定的访问压力。

**容灾手段：**

1.在我项目里，Redis只做加速；

挂掉时我们三层查询（Redis→ES→MySQL）保证可查，

恢复后重新回填缓存，数据不丢，用户体验略慢但可用。

2.生产环境用Redis集群+哨兵/主从，单点失败不影响整体,

发现挂掉后自动重启 + 主从切换.

3.Redis恢复后，可热加载热点数据（如热门商家评价列表）到缓存

# 14.Redis的过期时间是怎么设置的

在我这个项目中，我方便学习统一设置的是10秒，

但是在实际生产环境下，要根据不同商品的热度动态调整，

热门商品：1小时

一般商品：10分钟

冷门商品：1分钟

对于一些一直热门的商品可以设置异步刷新的策略，永不过期。

因为一直热门的商品他的访问量一直是很高的，如果过期了，

缓存击穿会导致大量请求打到MySQL中，数据库压力还是很大的。

或者可以给那些比较热门的商品添加随机偏移的策略（防止雪崩），就是过期时间加随机值。

# 15.kafka的作用？

kafka负责直接连接Canal和ES，如果ES发生故障，

此时也没有kafka的话，Canal会写入失败，

可能导致Canal自身报错甚至崩溃，整个链路瘫痪。

但加入Kafka后，Canal会继续写入Kafka，Kafka持久化到磁盘，

ES恢复后消费者服务会继续消费，故障完全隔离。

同时Kafka还能削峰缓冲，防止高峰期ES被打挂。

# 16.kafka挂了怎么办？如何防止？

**产生的问题：**

 **短期故障的情况** ：当Kafka无法连接时，

Canal会检测到连接失败，开始进行重试。

Canal内部有重试机制，会不断尝试重新连接Kafka，

通常配置为无限重试或者很大的重试次数。这个过程中，

来自MySQL的Binlog消息会堆积在Canal的内存缓冲队列里。

一旦Kafka恢复，Canal立即重新连接成功，

缓冲的消息全部发送到Kafka。这个过程是自动进行的，

不需要人工介入。整个系统会恢复正常，数据不会丢。

 **长期故障的情况：**。如果Kafka几个小时都没恢复也没有集群处理，

Canal的内存缓冲会越来越满。MySQL继续产生Binlog，

Canal继续读取Binlog，新的消息不断到来，

但无处堆放。最终，Canal的内存会耗尽，

程序out of memory，进程被杀死。此时Canvas停止运行。

但是，即使Canal进程崩溃，**数据仍然是安全的。**

因为canal挂了的话，**对于MySQL本身** 而言。数据永久存储在MySQL中，

这是源头，无论Canal和Kafka怎么样，MySQL的原始数据始终在。

 然后**MySQL的Binlog机制也保证数据安全** 。

MySQL会自动产生Binlog，默认保留7天。

只要Kafka和Canal故障发生在这7天内，

Binlog都还在磁盘上，没有任何丢失的风险。

Binlog文件保存在MySQL的数据目录，

格式是mysql-bin.000001、mysql-bin.000002这样的顺序文件。

 然后是**Canal的meta.dat** 。

Canal会定期记录它消费了MySQL Binlog的哪个位置，

也就是具体是哪个Binlog文件的第几个字节。

这个位置信息存储在meta.dat文件中。

当Canal重启时，它读取meta.dat，

知道自己上次消费到了哪里，然后从那个位置继续向后消费。

**容灾手段：**

**1.使用Kafka集群方式** 。例如一个broker（服务节点）故障，

其他broker继续工作，整个Kafka不会Down。

**2.监控和告警** 。实时监控Canal→Kafka的同步延迟，

一旦发现延迟超过阈值(比如5分钟)，立即告警通知运维。

**3.定期校验** 。每天凌晨运行数据对账脚本，

检查MySQL和ES的数据是否一致。

如有差异，自动触发补偿机制(重新同步)。

# 17.为啥要选kafka作为消息队列

1.我的项目选用 Kafka 作为消息队列确实是考虑了一些因素，

首先我查了资料，了解到Kafka 支持高吞吐量场景，

很适合电商评论这种高并发场景。

2.kafka有持久化和分区副本机制，保证消息不丢失，系统可靠性高。

3.Kafka 与其他的模块兼容性好，

比如我们用 Canal+Kafka 实现 MySQL 到 Elasticsearch 的数据同步，

这在业界是主流方案，RabbitMQ 和 RocketMQ 在这方面支持有限。

4.Kafka 社区活跃，文档丰富，横向扩展都很方便，

适合我们这种需要横向扩展的场景。

rabbitMQ他与一些大数据生态集成性不如kafka方便，

而且RabbitMQ吞吐量有限，不适合我们这种高并发电商场景，

RocketMQ虽然吞吐量高，但是但生态和兼容性不如kafka，

5.kafka缺点就是他的消息延迟略高rabbitMQ，但是对于我们这种

实时性要不高的评价系统而言可以接受。

综合考虑，Kafka 是最适合我们项目的消息队列选型

# 18.**Canal的作用**

Canal伪装成MySQL的从库，通过MySQL主从复制协议获取binlog，

这个原理和MySQL的主从复制原理差不多，

MySQL的主从复制原理是：

主库 → 产生binlog → 从库拉取binlog → 从库按照binlog回放

Canal的做法：

主库 → 产生binlog → Canal伪装成从库 → 解析binlog → 发送到Kafka

# 19.**Canal挂了怎么办？如何防止？**

**产生的问题：**

Canal实际上是我们从MySQL读Binlog的中间层，

它把数据库变更推给消息中间件（Kafka）。如果Canal挂了，

写数据库的动作依然正常，MySQL数据还是在；

但Canal不再把Binlog推进Kafka，ES就不会同步新数据。

也就是说，业务写入不受影响，只是ES数据会滞后。

**容灾手段：**

1.针对这个问题可以采用的是降级到Mysql，就是先走redis发现找不到，

去ES中找，发现也找不到，就去MySQL中找，虽然慢，但至少保证能查到数据。

2.通过 **Canal集群模式** （多数生产环境用）：

* 多个Canal实例
* 一个Leader在跑，其他做备份
* Leader挂掉，自动选举新Leader（zookeeper选主）

3.可以做数据恢复：数据是不会丢失的，因为MySQL的binlog会保留，

Canal挂了之后，MySQL继续产生binlog，

binlog文件保存在磁盘上（默认保留7天），

Canal会记录消费位置，Canal的meta.dat文件记录binlog的position信息，

Canal恢复后，从上次的位置继续消费，

他会先读取meta.dat，从上次的position继续消费，不会重复也不会丢失。

# 20.Zookeeper了解多少

Zookeeper是一个开源的分布式协调服务，

核心功能是提供**一致性协调、配置管理、分布式锁、选举**等。

它本质上是一个轻量级的键值层次存储系统，

支持 `ephemeral`和 `sequential`节点。”

“在Canal集群高可用中，Zookeeper主要做**主备选举**和 **状态心跳** ：

每个Canal实例在ZK里注册一个 `ephemeral（临时节点）和 sequential（顺序编号）`，

最小序号者为临时节点Leader。Leader负责消费MySQL binlog并写Kafka。

Leader挂掉后，节点自动失效，剩余节点重新选举最小序号，

新的Leader接管，实现快速故障转移。”

“这样就保证了Canal集群里只有一个主节点在跑，

避免多实例重复消费binlog、写重复消息。

这种选leader的机制很像raft选举的机制。

# 21.为什么用zookeeper不用raft？

**两者区别：**

Zookeeper更像“谁先占坑谁当Leader”，“挂了就重新选最先登记的人”。

Raft更像“大家投票选出最稳的人”，“要求日志一致、大家都达成共识

**选zookeeper原因：**

Canal的核心职责是 **把MySQL Binlog复制出来并推送到队列** ，

这本质上是一个**协调/选主**问题，不是状态机日志复制问题。

Canal只要一个Leader来做消费、写Kafka，Follower做备份即可。

Zookeeper的选主机制足够了，开销小，配置简单。

而Raft是设计来做 **强一致的日志复制与状态机一致性** ，

复杂度更高，代价更大，不适合Canal这种轻量选主场景。

# 24.gRPC他是如何定义的

它基于 Google 的 HTTP/2 协议，并使用 Proto Buffer作为数据协议，

相比 HTTP/1，它是二进制协议，比文本协议更高效；

支持  **多路复用** ，也就是一个连接上可以并行发送多个请求，

减少了传统 HTTP 协议中的连接开销；另外，它有  **头部压缩** ，

减少了请求和响应的大小，从而提高了网络传输的效率。

# 25.gRPC和http使用在什么场景下

rpc调用发生在微服务之间的函数调用过程中，http是用于和前端交互场景下。

grpc提供了更高效的、低延迟的服务间调用。而前端浏览器不直接支持 gRPC，

如果前端需要与 gRPC 服务交互，可以使用 **gRPC-Web** 这种技术，

它通过 HTTP 协议与浏览器兼容，并通过代理转发请求给后端的 gRPC 服务，

**HTTP** 通常使用 **JSON** 格式，JSON 在浏览器中有原生的支持，

且易于与 JavaScript 进行交互。因此，前端开发者通常更倾向于

使用**JSON** 格式的数据进行请求。

对于grpc来说，protobuf 的数据序列化与反序列化速度非常快，

gRPC 可以实现更低的通信延迟，对于需要高性能、高吞吐量的服务间通信是很重要的。

gRPC 支持  **多种编程语言** （包括 C++, Java, Go, Python, Ruby, Node.js 等），

并且通过 protobuf 可以自动生成客户端和服务器端的代码。

这使得不同技术栈之间的微服务能够高效地进行通信。

虽然 **HTTP/REST** 也有跨语言的支持，但是 **gRPC通过 `.proto` 文件自动生成代码，**

**这样会减少了跨语言通信的出错几率。**

# 26.http/1.1、http/2、http/3区别

* 基于TCP，采用文本协议，一问一答，支持长连接（Keep-Alive）。
* 最大缺点是“队头阻塞”：同一连接上请求必须按顺序处理，前面没返回，后面只能等。
* 适合简单的网页、接口请求，低并发场景。

**HTTP/2：**

* 还是基于TCP，但采用二进制分帧协议，支持多路复用（一个连接上可以并发处理多个请求）。
* 解决了队头阻塞，提高了并发性能，支持头部压缩（HPACK）。
* 适合高并发、微服务、接口通信、gRPC等场景。

**HTTP/3：**

* 底层不再用TCP，改用UDP+QUIC协议（解决队头阻塞），可以快速重连，彻底解决TCP级队头阻塞。
* 网络切换、弱网环境下表现更好，适合移动端、车联网、实时通信等场景。

我的理解就是比如去银行排队办业务，每个人都要按顺序来，

在HTTP/1.1里，前面的人没办完，后面的人只能干等。

HTTP/2改进了，大家可以同时递交材料，谁先处理完谁先走（多路复用）

HTTP/2其实解决了“排队等前面人”的问题，但底层丢包时大家还是一起卡。

HTTP/3让丢包只影响丢的那一份，其他人不受影响，效率最高。

发布链路是 CI 构建镜像、推仓库、更新镜像 tag，异常就快速回滚。
中间件方面优先用托管版 MySQL/Redis/Kafka/ES；

如果中间件必须自己搭，我才会放到 K8s 里跑。
这种像 MySQL、Redis、Kafka 这种有数据的服务，

我会用专门的方式部署，保证数据不丢、挂了能顶上。
同时把监控和告警都配好，出问题能第一时间发现。

迁移的时候我不会一次全搬，我会先搬业务服务这种好迁移的，

再搬数据库和消息队列这种风险高的。
先保证系统稳定可用，再慢慢做性能和成本优化。

* `Deployment`：
  管理“业务服务”的，比如Go 接口服务。
  它能让服务开多份（多副本），并且升级时一批批替换（滚动发布），不停机。
* 多副本：
  同一个服务跑 2~3 份以上。挂一份还有别的顶上。
* 滚动发布：
  新版本不是一次全换，而是先换一小部分，没问题再继续换，风险小。
* `readinessProbe`：
  “准备好没？”没准备好就不接流量。
* `livenessProbe`：
  “还活着没？”卡死了就自动重启。
* 资源限制：
  给每个服务设置 CPU/内存上限，防止一个服务把机器吃满。
* `HPA`：
  自动扩缩容。忙了就多开几份，闲了就少开几份。
* `Service`：
  服务访问入口。别的服务通过它来找到你，不用关心具体 Pod IP。
* `ConfigMap`：
  放普通配置（如端口、开关、地址）。
* `Secret`：
  放敏感信息（密码、密钥、token）。
* CI 构建镜像 -> 推仓库 -> 更新 tag：
  自动化发布流水线。代码提交后自动打包、上传、上线。
* 快速回滚：
  新版本有问题，一键切回旧版本。
* 托管版 MySQL/Redis/Kafka/ES：
  云厂商帮你运维，你少管机器和高可用，稳定性更高。
* `StatefulSet` + `PVC`：
  给“有状态服务”用的。
  `StatefulSet` 管实例身份和顺序，`PVC` 负责持久化磁盘（重启数据不丢）。
* 先无状态、再有状态：
  先上业务服务（好迁移、可快速回滚），
  再上数据库/消息队列这类中间件（复杂、风险高）。

工作点一：

我们系统里有一个用来大批量解析dlt 日志的接口，

原来前端发来的每次请求都要花十几秒去完成，体验很不好。

原来的业务流程是，先遍历一遍dlt文件夹里面的所有proto文件，

每当遍历到一个proto文件就会使用正则方式提取我们需要的关键字段，

并生成每个proto文件对应的service对象存在内存中，

这个对象包含了刚才正则提取的字段。第二次 遍历service对象，

每遍历一个对象就会读取一个mapping文件（这个文件是一个解析规则的文件），

然后会按照规则将service对象转化并写入json中，最后把json传给前端。

主管让我去进行优化，我分析后发现，

在第二次遍历service对象的过程中，每次处理service都要读一个mapping文件，

而且我们service对象的数量是很大的，所以磁盘io的次数非常多，

于是我的优化是预读一次mapping规则文件，

把里面的规则内容存在内存的大字典的键值对中，

然后后续遍历处理service只需要走内存就行了，

然后我用brono前端测试发post请求发现从11秒提升到了8秒左右。

第二个优化点是第一次遍历的时候通过正则处理文件的行为是串行的，

于是我使用了并发协程+sync.mutex锁来进行处理，

因为每一个service对象也都是存在了内存的map中，map本身也是不安全的，

他的扩容等机制会出现一些问题，例如并发写的时候可能会造成数据冲突，

所以我这里加了mutex锁，然后协程数量我调整的策略是，

根据用户电脑的内核数量来动态调整，因为协程他本质是用户级线程，

真正做到并行还是要根据内核数量来看，

不然一个内核无论开多少并发总时间是不变的，甚至更久。

工作点二：

我们有一个数据洞察模块，从数据库查出来的数据里经常有 NULL 值，

传到业务层处理时就会报错误，前端也会返回500错误。

这是因为我们这个项目是用go去重构原来的python项目，

在python中，如果数据库是空的，python的结构体字段也会默认是Nan，

但是go中我们的结构体的字段是数值类型的，数据库中如果是空的，

就会报错。一开始我的做法比较简单，就是在db层的结构体里用指针类型来接数据库字段，

空值他会变成nil，对于某些不参与数值计算的接口而言，

就比较适用，只需要在db层改动就可以了，

但是有些接口他还需要再业务层进行计算处理，nil是无法计算的，

于是我当时在业务层加了一个自定义转换函数去外包一下每个变量，

如果是nil，转为0进行处理，这样做的缺点就是，业务层大量外包这种函数，

耦合关系复杂。后来我做了重构，在DB 层就引入 sql.Null 这类可空类型做中间变量，

如果读到 NULL，就直接在这里转换成默认值（比如 0 或空字符串），再往上传。
这样就相当于在数据进入业务之前做了一层清洗，业务层拿到的永远是有效值，

代码也更干净稳定，不会因为脏数据而崩溃，而且做到了数据层和业务层的完全解耦。

工作点三：

我们系统要支持好几种日志文件的解析和展示，其中有一个解析log日志的模块，

需要解析成一个csv和两个txt文件保存在本地并用json的方式返回给前端供前端查询查看，

解析的业务逻辑已经做好了，我是负责后端给前端的接口的实现，

对 CSV 文件，因为主要是按时间查日志，我就只提取时间列做一个轻量索引，

前端可以很快地按时间定位，不用加载全部内容。

对 两种TXT 文件，则会把关键报错信息完整提取出来，一次性用json返回给前端展示。

这样完成了整个的接口开发与对接需求的任务。

工作点四：

我负责开发针对我们日志解析的MCP-server端以及cli工具的开发，

进而开发了主后端agent以及他支持的skills，可以使得用户在客户端调用我们的skills去自动完成日志解析成明文，

然后通过脚本高效提取异常错误信息，最后生成html的日志异常错误的报告。

### 3.为什么要本地化而不是中央服务器？

其实我们当初架构设计时，也考虑过是不是把文件全传到中央服务器去解析。

最终决定做本地化部署（类似边缘计算的模式），主要是出于两点考虑：

**1.规避网络带宽与大文件的传输**：

如果每次都要研发人员把这么大的文件通过网络上传到中央服务器，

光是 **上传的时间可能就要几十分钟** ，更不要提万一网络不稳定断点续传的麻烦。

让数据**‘就近计算’**，在本地直接解析，效率是最高的。

2.**算力成本的分散** ：

日志解析过程中，里面包含了大量的正则匹配和文件 IO，是非常消耗 CPU 的。

如果公司几百上千个开发人员同时在中央服务器上解日志，

服务器的 CPU 极易被打满，公司需要为此付出巨大的机器成本。

而采用本地化部署，本质上是 **白嫖了开发人员本地电脑**

**（比如高配的 Mac/Windows 工作站）的算力** ，大大节省了服务端的开销。

### 4.为什么用go重构python？

**1.分发与部署的问题**

因为Python 是解释型语言，如果要把它做成一个本地工具散发给所有开发人员用，

不管是要求每个人配 Python 环境，还是用 `PyInstaller` 这种工具打包，

都非常痛苦。打出来的包体积巨大、启动慢，

而且经常遇到各种操作系统的动态链接库依赖报错。

PyInstaller是一种**假编译（或者说捆绑式打包）**，

原理不是把代码编译成底层的机器码，而是**把整个 Python 解释器、**

**所有依赖库和我们的脚本当作一个压缩包**捆绑在一起。

用户每次双击运行，它都要在后台先解压出一个临时文件夹，

拉起解释器才能跑代码。这就导致启动非常慢，

而且即便是个小脚本，打出来的包也动辄大几十兆。

我们很多同事是用 Mac 办公，但也有一大批用 Windows 的同事。

在 Python 里，你很难在 Mac 电脑上去打包一个干净的 Windows `.exe` 程序，

而  **Go 是静态编译型语言** ，它可以直接交叉编译（`go build`）出一个

 **没有任何外部依赖的独立二进制文件（Single Binary）** 。

对于工具的分发来说简直是降维打击，

用户拿到压缩包，解压后双击就能直接运行。

并且在我的 Mac 电脑上，只需要一句 `GOOS=windows go build`

就能立刻打出一个 Windows 用户直接能运行的 exe，体验极好。

2.**性能与多核利用率的好处**

日志的大批量解析是典型的  **CPU 密集型任务** 。

Python 有  **GIL（全局解释器锁）** ，它的多线程其实是伪多线程，

同一时间只有一个线程在使用一个内核，所以要实现真正的并行，

需要开多进程，这样的话mapping规则文件就要复制多份，内存开销就很大，

其实，Python 传统上有三种并发方式：**多进程**、多线程、协程（asyncio）,

而这个协程其实还是单核并发机制，受制于GIL(主要为了解决网络 I/O 等待问题),

Python 3.13 的 No-GIL 目前还是实验性质，

生态里大量底层的 C 扩展库（甚至包括一些正则和底层处理库）,

在没有 GIL 保护后，还存在线程安全问题.

而 Go 天生支持高并发,他的GMP调度模型就是可以做到多核利用，

这也是我在优化接口时，**能够直接根据 `runtime.NumCPU()` 动态调整协程数，**

**把用户电脑的算力真正‘榨干’** ，

最终把解析时间从 11 秒压到 8 秒的核心原因，这在 Python 里是很难做到的。

3.**工程的可维护性高**

Python 的动态类型在小脚本时代很爽，但系统一大，

数据库里的脏数据（比如 NULL 或 NaN）传到业务层到处乱跑，

一旦报错连堆栈都很难找。换成 Go 这种强类型语言后，

只要结构体或 DB 层映射没对齐，编译期或反序列化时立马就暴露了，

逼着我们（比如我引入 `sql.Null` 做清洗）把脏数据拦截在最底层，

大大提高了工具运行的稳定性。

### 5.采用的什么协议进行抓包，为什么？

我们采用的是tcp协议，因为这些日志类型包括dlt日志文件

（里面是诸多二进制的proto文件）、

BLF文件（BLF 是 CAN/LIN 总线数据的二进制日志格式，适合高频数据采集）、

pcap文件（PCAP 则是网络通信的原始数据包，常用于以太网通信分析）。

我们后端软件远程抓包时，底层一般是通过 TCP/IP 协议进行通信。

### 6.为什么不用rpc协议，区别？

RPC协议（比如gRPC）本身就是用二进制格式传输数据，

效率高、序列化体积小，非常适合微服务之间的数据通讯。

但我们抓包场景和微服务场景有本质区别：

我们抓取的是完整的日志文件（比如DLT、BLF），

这些文件本身就是二进制，体积大，

通常直接用TCP/IP协议进行文件传输，保证数据完整和可靠。

虽然RPC（如gRPC）底层也是基于HTTP/2和TCP，

传输的是二进制流，但它的设计初衷是为了高效传递结构化的小数据，

比如接口参数、业务响应，适合微服务通信和远程调用。
对于我们抓包这种大文件传输场景，直接用TCP传文件更简单高效，

不需要额外做分片、流式处理。如果用RPC传大文件，

还要自己设计分片、断点续传等机制，反而复杂。

### 7.介绍一下你做的agent

我这个 agent 的背景是这样的：我们原来已经有一套给 Cursor 外部 agent 用的 MCP 能力，

比如 pcap、blf、idl 这些解析模块，Cursor 可以通过本地 MCP server 二进制文件直接调起来完成日志或协议解析。

但这种方式更偏开发侧，前端用户没法直接复用，他们也得去安装cursor去配置json，而且得把这些二进制文件安装到本地。

所以我们要在主后端里做一层 agent，

把原来面向 Cursor 的 MCP 能力往业务系统里收口。

这样前端用户可以直接通过自然语言访问主后端的 skills，

由 agent 的skill负责做意图理解、参数提取和能力编排，

再去调用后端内部的 MCP 模块或工具函数，

最终完成日志解析、基线准备、报表生成这些任务。

就是说，我做这个 agent 的核心目的是把原来只服务 Cursor 的外部工具能力，

统一接入到我们自己的后端系统里，让前端和外部 agent 两种入口的效果保持一致。

### 8.（被问到skills.md相关的可以参考这个）

我这个项目里的 skill 是工程化落地版，不是平台化声明式 skill，

它不是把 **SKILL.md** 直接交给平台在运行时就解释执行了，

他是把 skill 做成了工程里的一个任务工作流。

**所以SKILL.md** 在本项目中是能力说明以及边界定义的文件，真正执行是在 **runner.go** 里完成的。

运行时模型主要看的是代码里的 prompt 和底层工具描述。

就是说，我们这里的 skill 重点是，

把自然语言理解、参数补全、任务编排和底层 MCP 能力调用个给串起来了，

不是做了一个很重的平台化系统。

### 9.skills是怎么工作的，怎么进行的？mcp和skill的关系？

在我的项目里，skill是引用了mcp模块底层，**MCP** 和 **skill** 是分层关系。

**MCP** 更像底层工具能力层，负责具体执行，比如 **pcap**、**blf**、**idl** 解析；

**skill** 是 agent 层里面的，面向用户给的任务的工作流的封装。用户请求先进入 **agent** 接口，

再路由到具体 skill。SKILL.md 是给开发者看的说明文档，

skill/runner.go 里是 skill 的具体执行逻辑。

执行时，skill 会先判断是不是 **chat-only**，

再结合模型做意图提取，把自然语言转换成结构化参数，

然后做规则判断和参数补全，

最后它再去调用底层的 **MCP** 模块或者内部工具函数完成执行，

并把结果写入 session，支持后续追问。所以从我项目的角度看，

**skill** 本质上是在做自然语言理解、任务编排、工具调用和上下文管理，

（而 **MCP** 更偏底层能力提供）。

### 10.logparser的解析skill模块是怎么运作的？

在我这个项目里，**logparser** 这个 skill，是一个“日志诊断任务”的工作流封装。

他把“日志解析、异常提取、模型总结、报告生成”这几个步骤串起来了。

用户请求先进入 **agent** 层的 skill 接口。进入之后，

系统会先判断这次请求是不是 **chat-only**，

也就是用户是不是只是在继续追问，而不是重新发起一次日志分析。

如果只是聊天，就直接走会话回复，不会真的触发解析流程。

如果不是 **chat-only**，就会进入这个 skill 的 **runner函数**。

**runner** 会先做一层意图提取，也就是从用户的自然语言里拿到结构化参数，

比如日志路径、日志类型、用户真正想问的问题。

如果用户显式传了这些参数，就优先用显式参数；如果没传，就从自然语言里补出来。

接下来这个 skill 会真正开始执行分析流程。

它先调用项目里的底层日志解析能力，把原始日志转成结构化 JSON。

然后再从这个 JSON 里提取异常片段，

比如 error、exception、timeout 这类关键信号。拿到这些异常片段之后，

再把用户问题、日志摘要和异常信息一起交给大模型，让模型输出诊断总结，

比如主要异常、可能根因和建议排查步骤。

最后再把这些结果生成成一个 HTML 报告，方便用户查看。

流程结束后，系统还会把这次 skill 的执行结果写进 session，

包括日志路径、解析结果路径、报告路径和摘要。

这样用户后面如果继续追问，就可以基于这次分析结果继续聊。

### 11.rules 是怎么做的？

在我这个项目里，**rules** 不是单独做成规则引擎，

他是分散在 **agent skill**、**MCP** 接口约束和业务代码里一起工作的。
比如第一层是 prompt 规则，主要写在 skill 的 **runner.go** 里。

它会先告诉模型当前是一个意图提取器，只能输出固定 JSON 字段，

这样把用户自然语言先生成结构化参数。然后第二层是代码里的硬规则，

比如怎么判断 **chat-only**、显式参数和模型提取结果谁优先、

没有给版本时是否补默认值、要不要先准备 baseline，

这些都不是靠模型自己决定，而是代码强约束。
第三层是底层能力接口的规则，也就是 **MCP** 或内部模块本身对输入输出的要求。

skill 在上层负责任务编排，真正执行时还是要满足底层接口的参数约束和调用方式。
所以我这个项目里的 rule 实现思路不是“一个中心化规则引擎”，他是多层约束，

最终稳定性主要靠代码兜底。

比如在 **log-error-report** 这个 skill 里，**chat-only** 判断就是代码规则；

日志路径、日志类型和问题提取里有 prompt 规则；

而真正执行日志解析、异常提取和报告生成时，

又要遵守底层函数和模块的输入输出约束。

### 12.如何防范恶意输入或者注入？

我这个项目的安全防护更偏 **工程约束型** ，不是做了一个完整的安全中台，

因为它本身是本地化后端场景，不是公网多租户服务。

我的核心思路是尽量 **不要让大模型直接决定执行什么命令** ，而是把执行边界收死。

第一层是  **skill 和 MCP 的能力边界** 。我暴露的不是一个通用执行器，

而是固定的几个 skill 和固定的几个 tool，比如 **pcap**、**blf**、**idl**、日志分析。

用户就算恶意提示，也只能落到这些受控能力里，不能让模型随便生成 bash 或 Python 去执行。

第二层是  **结构化参数约束** 。在 **runner.go** 里我会先让模型做意图提取，

但要求它只返回固定 JSON 字段，比如文件路径、版本号、输出路径这些，

后面代码再做解析和校验。也就是说，LLM 只负责理解语义，不直接拿它生成的自然语言去执行。

第三层是  **代码里的硬控制** 。像主后端这条链路，我后来专门改成直接调项目内函数，

而不是走 CLI；独立 MCP server 才走 CLI。如果需要起子进程，

也都是 **exec.Command** 这种固定二进制 + 固定参数，不走 shell 拼接，所以能避免典型的命令注入。

第四层是  **输入校验** 。像文件路径、版本号这些，

在 handler 和底层 parser 里都有必填校验、路径存在性检查、部分场景下的绝对路径检查，

不会因为模型说了一段话就直接执行。

如果是公网项目的话，我也了解过。

比如我们可以做一些**权限控制** 。
不是谁都能调用所有 skill，不同用户、不同角色、不同租户，

能用的能力和数据范围都要分开。比如有人只能做日志分析，有人才能跑更高风险的解析能力。

然后可以做**租户隔离和数据隔离** 。
如果是公网服务，就不能像本地项目这样默认信任输入了。每个用户上传的日志、pcap、分析结果、session，都要隔开，不能串数据，也不能因为一个 session key 猜出来就拿到别人的结果。

第三个是 **执行环境隔离** 。
如果底层真的要跑解析程序或者子进程，我不会直接让它跑在主服务进程里，而是会放到单独的沙箱环境里，比如容器、隔离 worker、资源受限的执行环境。这样就算有恶意输入，影响面也不会直接打到主服务。

第四个是 **输入校验和限流** 。
公网下恶意输入会更多，所以除了参数校验，还要防止超大文件、恶意路径、频繁调用、批量刷接口这些问题。比如要做文件大小限制、接口限流、并发控制，还有任务超时。

第五个是 **审计和监控** 。
公网服务一定要能追踪是谁、在什么时间、发了什么请求、调用了哪个 skill、最后执行了什么。这样出了问题才能排查，也方便做风控。

第六个是 **大模型本身的防护** 。
比如 prompt injection、越权调用工具、诱导模型泄露系统信息，这些在公网下会更常见。所以我会把模型能看到的系统提示、能调用的工具、返回给用户的数据都做边界控制，不会让模型自己无限制决定。

### 13.skill后续改进方案

我觉得这个项目在初期这样设计是合理的，

因为它的优点很明显，就是开发快、逻辑稳，而且整体比较好控制。
但如果后面 skill 变多了，问题也会慢慢出来。

比如现在 skill 路由基本还是写死的，用户很多时候得自己选要走哪个 skill；

另外一些规则分散在不同的 **runner** 里，后面 skill 一多，

维护成本会越来越高；再往后如果用户的需求不是单一步骤，

而是要几个 skill 串起来一起做，那现在这套方式也会比较别扭。

所以我也考虑后续去演进，
比如我会先把现有 skill 做得更统一一点，

把每个 skill 的名称、输入输出、执行入口这些都收拢起来，

做成一套统一注册的方式。这样以后新增一个 skill，

就不用到处改路由、改 handler、改很多分散逻辑。

后续的话我也会在 **agent** 前面加一层意图路由。也就是用户先说需求，

系统先判断他更适合走哪个 skill，或者是不是需要几个 skill 配合，

而不是一开始就让用户自己选。
但这个阶段我不会完全交给大模型自由决定，

比如可以明显的场景先用规则判断，

不那么明确的再让模型辅助识别，这样会更稳。

如果后面业务再复杂一些，我会考虑让大模型进一步参与 skill 的选择和任务拆解。
但不能完全放开让模型随便选，也会给它一些边界。

比如只让它在我们定义好的 skill 范围里选，传参也必须符合固定格式，

执行失败时要能退回到保守方案，整个过程也要能记录下来，

方便后面排查问题。也比较灵活吧。

### 14.openclaw的理解

小龙虾我对他的理解其实就是对比了其他的一些智能体，比如cursor。我先说一下他俩的一些差异。

Cursor 本质上是VS Code 进程里的一个子模块 。它的所有能力，都被限制在 IDE 的沙盒环境里，架构上依赖于编辑器的插件 API。它能做的事，比如补全代码、分析语法树，都离不开编辑器这个宿主。

而小龙虾是一个**独立的系统级进程 。它在架构上是 **顶层设计** ，直接对接操作系统内核，拥有系统级的权限。这意味着它在架构上不依赖任何工具，它自己就是一个 **执行平台** 。

**其次，是能力扩展的架构设计不同。**

Cursor 的能力是功能内嵌** 的。比如代码重构、解释逻辑，这些功能是写死在代码逻辑里的，要加新功能就得改源码。

而小龙虾采用的是松耦合的插件化架构** 。它的核心引擎和具体能力是完全解耦的，所有功能（比如操作文件、执行 Shell、访问数据库）都封装成独立的 `Skill` 插件。只要实现统一接口，就能无限扩展。
